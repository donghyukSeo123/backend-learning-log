# [배포 학습] Docker 볼륨 마운트를 통한 파일 영구 보존 및 컨테이너 간 네트워크 연동

이 가이드에서는 Docker 배포 환경에서 컨테이너 업데이트(재빌드) 시 발생하는 **파일 유실 문제(Data Persistence)**와, 격리된 컨테이너 간의 **네트워크 연결 문제(Container Networking)**를 해결하고 이해하기 위한 핵심 개념을 공부합니다.

---

## 1. 컨테이너 간 네트워크 연동 (파이썬 AI 서버 호출 문제)

### ① 문제 상황 (localhost 호출 실패)
기존 스프링 부트(백엔드) 소스코드에는 파이썬 FastAPI 서버 주소가 아래와 같이 하드코딩되어 있었습니다.
```java
String pythonUrl = "http://localhost:8000/generate-content";
```
*   **원인**: 로컬 개발 PC에서는 백엔드와 파이썬 서버가 같은 Windows OS(동일한 localhost) 안에서 구동되므로 문제가 없습니다.
*   **한계**: 하지만 **Docker Compose** 배포 환경에서는 각각의 서비스(`insajang-backend`, `insajang-python`)가 자신만의 독립된 가상 네트워크 IP를 할당받아 별도의 컨테이너로 실행됩니다. 따라서 백엔드 컨테이너 내부에서 `localhost`를 호출하면, 파이썬 컨테이너가 아닌 **백엔드 컨테이너 자신**을 가리키게 되어 연결 오류(`Connection Refused`)가 발생합니다.

### ② 해결 방법 (환경변수화 및 도커 네트워크 명칭 사용)
Docker Compose 환경에서는 동일한 compose 파일 아래 묶인 컨테이너끼리 **컨테이너 이름(서비스 이름)**을 도메인 주소(DNS)처럼 사용하여 통신할 수 있습니다.

1.  **백엔드 설정 파일 분리 (`application.properties`)**
    ```properties
    app.python-url=${APP_PYTHON_URL:http://localhost:8000}
    ```
    *   기본값은 로컬용 `http://localhost:8000`으로 놔두어 로컬 개발 편의성을 유지합니다.
2.  **백엔드 소스코드 리팩토링 (`ContentService.java`)**
    ```java
    @Value("${app.python-url}")
    private String pythonUrl;
    
    // ...
    String targetUrl = pythonUrl + "/generate-content";
    ```
3.  **Docker Compose 환경변수 및 종속성 지정 (`docker-compose.yml`)**
    ```yaml
    insajang-backend:
      environment:
        - APP_PYTHON_URL=http://insajang-python:8000
      depends_on:
        - insajang-db
        - insajang-python
    ```
    *   `insajang-backend`가 `insajang-python` 컨테이너 이름으로 파이썬 AI 서비스를 정상 추적 및 호출하도록 도커 가상 도메인을 주입합니다.

## 2. Docker 볼륨 마운트 (파일 영구 보존)

### ① 문제 상황 (배포 시 업로드 파일 유실)
스프링 부트 백엔드는 AI 모델을 통해 생성한 이미지 파일 등을 `uploads/` 폴더 내에 저장합니다.
*   **원인**: 볼륨 연동이 없을 경우, 파일은 컨테이너 내부 가상 경로인 `/app/uploads`에 저장됩니다.
*   **한계**: Docker 컨테이너는 일회성 객체이므로 코드 수정 후 깃허브 푸시를 통한 자동 배포(컨테이너 삭제 후 재생성)가 일어날 때마다 **컨테이너 내부에 쌓였던 이미지 파일들이 모두 영구 삭제**됩니다.

### ② 해결 방법 (바인드 마운트 설정)
호스트(AWS EC2 가상머신)의 실제 물리 디렉토리와 컨테이너 내부의 디렉토리를 연결하여 데이터가 호스트 디스크에 안전하게 유지되도록 설정합니다.

1.  **Docker Compose 볼륨 설정 (`docker-compose.yml`)**
    ```yaml
    insajang-backend:
      volumes:
        - ./uploads:/app/uploads
    ```
    *   호스트의 `./uploads` 폴더(서버의 `~/Insajang/uploads`)를 컨테이너 내부의 `/app/uploads`와 동기화시킵니다.
    *   이제 백엔드 컨테이너가 재시작되거나 삭제 후 새로 생성되어도, 호스트의 `./uploads` 폴더에 물리 파일이 보존되어 있으므로 소실 걱정이 없습니다.

## 3. Nginx 업로드 파일 서빙 경로 연동

### ① 문제 상황 (이미지 엑스박스 출력 에러)
AWS 보안 그룹 설정상 백엔드가 구동 중인 `8080` 포트는 외부에서 접근이 차단되어 있고, 오직 Nginx의 `80` 포트(HTTP)만 대외적으로 열려 있습니다.
*   웹 브라우저가 이미지(`http://3.39.22.29/uploads/파일명.png`)를 요청하면, 포트 80을 수신하는 Nginx가 이 요청을 처리하게 됩니다.
*   하지만 기본 Nginx 설정 파일에는 `/uploads/` 경로에 대한 대리 처리(Proxy) 규칙이 누락되어 있어 브라우저는 이미지를 찾지 못하고 404 에러를 반환합니다.

### ② 해결 방법 (Nginx Reverse Proxy 설정 추가)
Nginx가 프론트엔드 정적 파일 서빙 외에도 `/uploads/`로 유입되는 이미지 파일을 백엔드 컨테이너로 포워딩할 수 있게 프록시 패스 설정을 구성합니다.

1.  **Nginx 구성 업데이트 (`nginx.conf`)**
    ```nginx
    # 업로드 이미지 요청 프록시 우회
    location /uploads/ {
        proxy_pass http://insajang-backend:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
    ```
    *   `/uploads/`로 시작하는 모든 이미지 요청을 백엔드 스프링 부트 서버의 `8080` 포트로 돌려주며, 백엔드 서버의 리소스 핸들러가 해당 요청을 확인하여 실제 물리 파일(`/app/uploads/파일명.png`)을 클라이언트 브라우저로 응답해 줍니다.
