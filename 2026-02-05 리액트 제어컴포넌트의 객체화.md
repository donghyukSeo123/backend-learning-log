# 2026-02-05 – 인사장프로젝트: 상태 관리의 한계 체감 및 객체화 전환 예고

## 오늘 느낀 불편함 (Pain Point)
- **useState 남발**: 회원가입 필드마다 `useState`를 따로 만드니 코드가 지저분하고 관리가 안 됨을 느낌.
- **로직 파편화**: 데이터가 분산되어 있어 나중에 백엔드로 보낼 때 하나하나 수집하기가 매우 번거로움.
- **코드 중복**: 비슷한 형태의 `onChange` 함수를 필드 개수만큼 만드는 비효율을 확인함.

## 해결을 위해 공부한 것
- **State Objectification**: 여러 상태를 하나의 `{}` 객체로 묶어 관리하는 구조적 설계법.
- **Single Source of Truth**: 데이터를 객체 하나로 통합하여 데이터의 일관성을 유지하는 방식.
- **Spread Operator (`...`)**: 리액트의 불변성을 지키면서 특정 값만 안전하게 업데이트하는 기법.

---

## 🧠 오늘의 깨달음

### 🚫 개별 useState의 한계
- **기존 방식**: 필드가 늘어날수록 선언부와 핸들러가 기하급수적으로 늘어나 가독성이 떨어짐.
- **문제**: 전체적인 데이터 흐름을 한눈에 파악하기 힘듦.

### ✅ 제어 컴포넌트의 객체화 (Next Step)
- **개선 방식**: `formData`라는 바구니 하나에 모든 입력값을 담아 관리.
- **기대 효과**: 코드가 획기적으로 줄어들고, 백엔드 전송 시에도 객체 그대로 전달 가능하여 편리함.



### ⚡ 통합 이벤트 핸들러의 원리
- 입력 태그의 `name` 속성과 객체의 `key`를 일치시켜, 단 하나의 함수로 모든 입력을 처리하는 메커니즘을 파악함.

---

## 🚀 다음 페이지 적용 가이드
1. **상태 통합**: 가입 정보 데이터를 `formData` 객체로 즉시 전환하여 코드 다이어트 실시.
2. **함수 최적화**: 다수의 핸들러를 `[e.target.name]: value`를 활용한 공통 핸들러 1개로 통합.
3. **불변성 실천**: 전개 연산자(`...`)를 사용하여 기존 데이터를 보존하며 상태를 업데이트하는 로직 구현.
