## 1. 컨텐츠 생성 및 저장 프로세스 구조 변경 (Refactoring)

기존의 단일 단계 저장 방식에서 데이터 무결성과 사용자 편집 자율성을 보장하기 위해 **'선(先) 로그 저장, 후(後) 최종 확정'** 구조로 프로세스를 전면 개편하였습니다.

---

## 2. 기존 방식 vs 변경 방식 비교

| 구분 | 기존 방식 (Direct Save) | 변경 방식 (Log-based Save) |
| :--- | :--- | :--- |
| **프로세스** | AI 응답 즉시 `Content` 테이블 저장 | AI 응답 시 `ContentLog` 테이블 임시 저장 |
| **사용자 개입** | 생성된 즉시 DB 저장 (수정 전 데이터) | 사용자 편집 및 검토 후 최종 저장 선택 |
| **데이터 정합성** | 불필요한 AI 생성물까지 본 테이블에 누적 | 확정된 양질의 컨텐츠만 본 테이블에 유지 |
| **장점** | 로직이 단순함 | 본 테이블의 데이터 오염 방지 및 편집 편의성 증대 |

---

## 3. 세부 기술 구조 (Implementation Detail)

### 3.1 [Step 1] AI 생성 및 로그 기록 (Temporary Log)
- **과정**: 사용자가 컨텐츠 생성을 요청하면 파이썬 AI 서버(FastAPI)와 통신하여 결과를 수신합니다.
- **기술적 처리**: 
    - AI가 응답한 데이터(Raw Data)를 `ContentLog` 엔티티에 매핑합니다.
    - 이때, `projectId`와 `userId`를 함께 기록하여 추적성을 확보합니다.
    - 리액트(FE)에는 생성된 텍스트와 함께 `log_id`를 반환하여 이후 단계의 참조 키로 활용합니다.

### 3.2 [Step 2] 편집 및 사용자 검토 (Edit Phase)
- **과정**: 리액트 화면에서 사용자가 AI가 생성한 제목과 본문을 직접 수정합니다.
- **기술적 처리**: 
    - 최종 저장 전까지는 DB의 본 테이블에 영향을 주지 않으며, 클라이언트 사이드에서 자유로운 편집이 가능합니다.

### 3.3 [Step 3] 최종 확정 및 본 테이블 저장 (Final Commit)
- **과정**: 사용자가 '최종 저장' 버튼을 누르면 편집된 최종본이 본 테이블(`Content`)에 기록됩니다.
- **기술적 처리**:
    - `log_id`를 기반으로 로그 데이터의 상태를 갱신하거나 연관 관계를 맺어 데이터의 생성 경로를 증명합니다.
    - 최종적으로 정제된 데이터만 서비스 운영 데이터로 활용됩니다.

---

## 4. 구조 변경에 따른 기대 효과
- **DB 최적화**: 무분별한 AI 생성 호출로 인한 불필요한 데이터가 메인 테이블에 쌓이는 것을 방지합니다.
- **사용자 경험(UX) 개선**: AI 결과물을 그대로 사용하는 것이 아니라, 사용자가 직접 2차 편집할 수 있는 유연한 흐름을 제공합니다.
- **비용 관리**: 각 로그를 기록함으로써 AI API 사용량 추적 및 통계 분석이 용이해졌습니다.
