# [배포 학습 - Part 2] Spring Boot 백엔드 (`insajang-backend`) 멀티스테이지 빌드

이 파트에서는 컴파일 도구가 필요한 Java/Gradle 환경의 Spring Boot 프로젝트를 배포 용량 최적화 기법인 **멀티스테이지 빌드**를 통해 가볍게 패키징하는 원리를 학습합니다.

---

## 1. Dockerfile 코드 분석
[Dockerfile](file:///c:/workspace/Insajang/insajang-backend/Dockerfile)

```dockerfile
# 1단계: 빌드 스테이지 (Gradle 빌드 도구와 JDK 17 포함 이미지 사용)
FROM gradle:8-jdk17 AS build
WORKDIR /app

# 빌드 설정 파일 및 소스코드 복사
COPY build.gradle settings.gradle ./
COPY src ./src

# Gradle을 사용하여 JAR 패키지 빌드 (테스트는 스킵하여 시간 단축)
RUN gradle build -x test --no-daemon

# 2단계: 실행 스테이지 (가벼운 JRE 17 이미지 사용)
FROM openjdk:17-slim
WORKDIR /app

# 1단계 빌드 결과물 중 JAR 파일만 실행 환경으로 가져옴
COPY --from=build /app/build/libs/*-SNAPSHOT.jar app.jar

# 백엔드가 사용하는 8080 포트 오픈
EXPOSE 8080

# 컨테이너 실행 시 Spring Boot 기동
ENTRYPOINT ["java", "-jar", "app.jar"]
```

## 2. 핵심 이론 및 원리 학습

### ① 멀티스테이지 빌드 (Multi-stage Build)의 필요성
*   자바 프로젝트를 컴파일하고 패키징하려면 **JDK(Java Development Kit)**와 빌드 툴인 **Gradle**이 필수적입니다. 이 두 개가 포함된 이미지는 매우 무겁습니다 (800MB 이상).
*   하지만 이미 완성된 `app.jar`를 실행할 때는 컴파일러가 필요 없으며, 실행 전용 엔진인 **JRE(Java Runtime Environment)**만 있으면 충분합니다 (`openjdk:17-slim` 등, 약 200MB).
*   **원리**: 1단계(`AS build`)에서 무거운 JDK 기반으로 컴파일을 끝마치고 JAR를 만듭니다. 그 다음 2단계에서 가볍고 날씬한 JRE 이미지를 시작하고, 1단계에서 빌드된 파일(`--from=build`)만 쏙 빼와서 실행합니다. 1단계의 무거웠던 컴파일 환경은 빌드 작업 종료와 동시에 흔적도 없이 사라져서 최종 서버 운영용 이미지는 200MB대로 깔끔하게 다이어트됩니다.

### ② `gradle build -x test --no-daemon` 옵션 분석
*   **`-x test`**: 빌드 과정 중 JUnit 테스트 실행을 제외합니다. 빌드 시점에는 가상 컨테이너 내부에 PostgreSQL 서버가 연결되어 있지 않은 상태이기 때문에, 테스트를 건너뛰지 않으면 DB 커넥션 예외가 터지며 빌드 자체가 멈추게 됩니다.
*   **`--no-daemon`**: Gradle은 기본적으로 컴퓨터 메모리에 데몬 프로세스를 상주시킨 뒤 백그라운드에서 다음 빌드를 기다립니다. 하지만 일회성 빌드가 끝나면 증발할 도커 컨테이너 내부이므로, 쓸데없이 메모리만 차제하는 데몬 기능을 명시적으로 꺼주어야 합니다.

### ③ `ENTRYPOINT` vs `CMD`
*   `CMD`는 컨테이너 시작 시 실행되는 명령을 지정하지만, 도커 실행 시 파라미터가 들어오면 쉽게 무시/덮어쓰기 됩니다.
*   `ENTRYPOINT`는 해당 컨테이너가 실행 파일(Executable)처럼 작동하도록 변경 불가능한 고정 기동 명령어를 지정합니다. 스프링부트 서버는 반드시 자바 런타임으로 JAR를 구동해야 하므로 `ENTRYPOINT`가 안전합니다.
