# [TIL] 단일 AWS EC2 + Docker Compose를 이용한 인사장(Insajang) 서비스 일괄 배포 및 연동

- **작성일**: 2026-06-09
- **주제**: Docker, Docker Compose, Nginx Reverse Proxy를 활용한 풀스택 서비스(React + Spring Boot + FastAPI + PostgreSQL)의 단일 EC2 인스턴스 배포 및 연동 정리

프로젝트를 단일 AWS EC2 인스턴스 환경에 무료(프리티어)로 안전하게 배포하고, 배포 후 발생할 수 있는 CORS 및 보안 위협을 컨테이너 네트워크 아키텍처로 해결하는 설계와 구현 과정을 정리한다.

## 1. 배포 아키텍처 및 마주한 문제 상황 (Architecture & Trouble)

### ① 배포 네트워크 아키텍처 설계
하나의 가상 서버(EC2) 안에서 여러 개의 서비스를 격리하여 안전하게 실행하고 통신을 최적화하기 위해 아래와 같이 컨테이너 네트워크를 조율했다.

```mermaid
graph TD
    Client[사용자 브라우저] -->|Port 80 HTTP| Nginx[insajang-frontend Nginx]
    
    subgraph Docker Internal Network (Bridge)
        Nginx -->|정적 리소스 반환| React[React Build Assets]
        Nginx -->|/api/* 요청 프록시 우회| Spring[insajang-backend Spring Boot:8080]
        Spring -->|내부 통신| Python[insajang-python FastAPI:8000]
        Spring -->|내부 통신| DB[(insajang-db PostgreSQL:5432)]
    end
```

### ② 마주한 설계적 과제 및 이슈 상황

*   **포트 다중 노출 문제**: 각각의 컨테이너(React 3000, Spring Boot 8080, FastAPI 8000, PG 5432)를 개별 포트로 노출하면 방화벽(보안 그룹)에 구멍이 숭숭 뚫려 보안상 매우 취약해진다.
*   **배포 환경의 CORS 에러**: 로컬 개발과 달리 퍼블릭 IP/도메인을 타고 들어올 때 프론트엔드 포트(80)와 백엔드 포트(8080)가 다르면 브라우저 보안 정책(CORS)으로 인해 통신이 차단된다.
*   **환경 변수 및 API Key 누출 위험**: OpenAI나 Gemini API Key, 메일 계정 비밀번호가 스프링부트나 파이썬 코드에 하드코딩되면 깃허브를 통해 외부에 노출되는 보안 위협이 생긴다.
*   **AWS 프리티어(t2/t3.micro) 자원 한계**: t2.micro 서버는 **RAM이 1GB**에 불과하다. 이 좁은 메모리 안에 빌드 도구와 컴파일러를 다 설치해 빌드하면 서버가 금방 먹통이 되거나 OOM(Out of Memory)으로 사망한다.

## 2. 해결 방법 및 배운 점 (Shooting & Learnt)

### ① Nginx 리버스 프록시를 통한 CORS의 근본적 차단
*   **해결**: 프론트엔드 이미지 내부에 **Nginx 웹서버**를 탑재하여 외부에는 오직 **HTTP 기본 포트(80번)** 하나만 열어둔다.
*   **작동**: 
    1. 사용자가 접속하면 Nginx가 리액트 정적 파일(HTML/JS)을 즉시 화면에 띄운다.
    2. 리액트에서 `/api/...`로 시작하는 상대 경로로 API를 호출하면, Nginx가 이 요청을 받아서 도커 내부 네트워크의 백엔드 컨테이너(`http://insajang-backend:8080`)로 백핸드로 넘겨준다.
*   **결과**: 브라우저는 프론트와 백엔드가 동일 주소(Port 80)에 존재하는 것으로 판단하여 **CORS 설정이 없어도 완벽히 에러를 피해 간다.**

### ② Dockerfile 멀티스테이지 빌드를 통한 이미지 다이어트
*   **해결**: 빌드용 무거운 이미지(Gradle/JDK, Node.js)와 실행용 가벼운 이미지(OpenJDK-slim, Nginx-alpine)를 하나의 Dockerfile 안에 세그먼트별로 분리해서 작성했다.
*   **결과**: 컴파일러와 빌드 찌꺼기(node_modules, gradle 캐시)는 1단계에서 모두 버리고, 최종 기동용 파일(React HTML/JS, Backend JAR)만 가벼운 실행 파일로 추출하여 디스크 용량과 빌드 메모리 소모를 극대화해 줄였다.

### ③ 도커 컴포즈(Docker Compose)의 가상 DNS 네트워크 및 보안
*   **해결**: 컴포즈가 가동될 때 생성되는 기본 네트워크망 안에서는 **컨테이너 이름이 곧 도메인 주소** 역할을 한다. 
*   **보안**: `insajang-backend`와 `insajang-python`은 외부 호스트 포트 매핑(`ports`)을 제거하여 외부 침입을 완벽히 차단하고, 내부에서만 서로의 컨테이너 이름(예: `insajang-db:5432`, `insajang-python:8000`)으로 안전하게 무전 통신하도록 구성했다.

### ④ `.env`를 통한 환경 변수 자동 바인딩
*   **해결**: 도커 컴포즈 구동 경로에 `.env` 파일을 두면 자동으로 파일의 설정 값을 로드하여 `docker-compose.yml` 내의 `${VARIABLE}`을 치환해준다. 깃허브에는 `.env.example`만 업로드하여 소스코드 내부의 민감 데이터를 완벽히 숨겼다.

## 3. 핵심 배포 코드 구성 정리

### ① [insajang-frontend/Dockerfile](file:///c:/workspace/Insajang/insajang-frontend/Dockerfile) (리액트 빌드 및 Nginx 결합)
```dockerfile
# 1단계: Node.js로 리액트 소스코드 번들 빌드
FROM node:18-slim AS build
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build

# 2단계: 가벼운 Nginx 서버에 빌드된 파일만 올려 배포 이미지 생성
FROM nginx:alpine
COPY --from=build /app/build /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

### ② [insajang-frontend/nginx.conf](file:///c:/workspace/Insajang/insajang-frontend/nginx.conf) (SPA 라우팅 및 프록시)
```nginx
server {
    listen 80;
    server_name localhost;

    # React 정적 파일 서빙 및 SPA 새로고침 시 404 방지
    location / {
        root /usr/share/nginx/html;
        index index.html index.htm;
        try_files $uri $uri/ /index.html;
    }

    # API 요청 프록시 우회
    location /api/ {
        proxy_pass http://insajang-backend:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

### ③ [insajang-backend/Dockerfile](file:///c:/workspace/Insajang/insajang-backend/Dockerfile) (스프링부트 멀티스테이지)
```dockerfile
# 1단계: Gradle 환경에서 소스 컴파일 및 JAR 생성 (테스트 스킵 및 데몬 미사용으로 빌드속도 개선)
FROM gradle:8-jdk17 AS build
WORKDIR /app
COPY build.gradle settings.gradle ./
COPY src ./src
RUN gradle build -x test --no-daemon

# 2단계: 최적화된 자바 런타임(JRE) 환경에서 빌드된 JAR만 가동
FROM openjdk:17-slim
WORKDIR /app
COPY --from=build /app/build/libs/*-SNAPSHOT.jar app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### ④ [docker-compose.yml](file:///c:/workspace/Insajang/docker-compose.yml) (컨테이너 오케스트레이션 설계)
```yaml
version: '3.8'

services:
  # 데이터베이스: 로컬 DB 툴 접속 테스트를 위해 5432 포트를 열었으며 볼륨 마운트로 데이터 영구보존
  insajang-db:
    image: postgres:15-alpine
    container_name: insajang-db
    ports:
      - "5432:5432"
    environment:
      POSTGRES_DB: insajang
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: "1234"
    volumes:
      - postgres_data:/var/lib/postgresql/data
    restart: always

  # 파이썬 AI: 포트 노출 없음(보안), 환경 변수로 API Key 주입
  insajang-python:
    build:
      context: ./insajang-python
      dockerfile: Dockerfile
    container_name: insajang-python
    environment:
      - GEMINI_API_KEY=${GEMINI_API_KEY}
    restart: always

  # 백엔드: 내부 도메인 주소로 DB 접근, 메일 계정은 환경변수 주입
  insajang-backend:
    build:
      context: ./insajang-backend
      dockerfile: Dockerfile
    container_name: insajang-backend
    environment:
      - SPRING_DATASOURCE_URL=jdbc:postgresql://insajang-db:5432/insajang
      - SPRING_DATASOURCE_USERNAME=postgres
      - SPRING_DATASOURCE_PASSWORD=1234
      - MAIL_USERNAME=${MAIL_USERNAME}
      - MAIL_PASSWORD=${MAIL_PASSWORD}
    depends_on:
      - insajang-db
    restart: always

  # 프론트엔드: 외부 포트 80만 직접 개방하여 모든 클라이언트 트래픽 진입점 역할 수행
  insajang-frontend:
    build:
      context: ./insajang-frontend
      dockerfile: Dockerfile
    container_name: insajang-frontend
    ports:
      - "80:80"
    depends_on:
      - insajang-backend
    restart: always

volumes:
  postgres_data:
```

## 4. 실전 AWS 배포 가이드 요약 (Action Plan)

서버 론칭 단계가 완료되면 Ubuntu 원격 세션 내에서 아래의 단계별 커맨드로 배포를 마스터한다.

### Step 1. 가상 메모리(Swap) 설정 (1GB RAM 보완)
프리티어 서버 빌드 중 멈추는 현상을 막기 위해 HDD/SSD를 메모리처럼 빌려 쓰는 가상 메모리(2GB)를 설정한다.
```bash
sudo fallocate -l 2G /swapfile
sudo chmod 600 /swapfile
sudo mkswap /swapfile
sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
```

### Step 2. 서버에 Docker & Docker Compose 설치
```bash
sudo apt-get update -y
sudo apt-get install -y apt-transport-https ca-certificates curl gnupg lsb-release
sudo mkdir -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
sudo apt-get update -y
sudo apt-get install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin
sudo usermod -aG docker $USER
exit # 재로그인 필수!
```

### Step 3. 소스코드 복사 및 빌드 실행
```bash
git clone [본인의 깃허브 저장소 주소]
cd Insajang
cp .env.example .env
nano .env # 실제 Gemini Key 및 메일 정보 기입
docker compose up --build -d
```

### Step 4. 배포 후 동작 검증 및 문제 해결
*   컨테이너 상태 검사: `docker compose ps`
*   특정 컨테이너 실시간 로그 추적: `docker compose logs -f [서비스이름]` (예: `docker compose logs -f insajang-backend`)
*   컨테이너 내부 쉘 접속: `docker exec -it [컨테이너이름] /bin/sh`
