# [배포 학습 - Part 1] Python AI 서비스 (`insajang-python`) 도커라이징

이 파트에서는 파이썬 기반의 FastAPI AI 서비스(`insajang-python`)를 Docker 이미지로 만들 때 적용된 핵심 기술과 설정들의 상세 원리를 공부합니다.

---

## 1. Dockerfile 코드 분석
[Dockerfile](file:///c:/workspace/Insajang/insajang-python/Dockerfile)

```dockerfile
# 1. 파이썬 경량화 이미지를 기반으로 설정
FROM python:3.10-slim

# 2. 컨테이너 내부의 작업 디렉토리를 /app으로 설정
WORKDIR /app

# 3. 패키지 설치 목록 파일 복사 및 설치 (캐시 미저장으로 이미지 최적화)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 4. 소스코드 전체를 컨테이너 안으로 복사
COPY . .

# 5. FastAPI가 사용할 8000번 포트 오픈
EXPOSE 8000

# 6. FastAPI 서버 구동 명령 실행
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

## 2. 핵심 이론 및 원리 학습

### ① Base Image: `python:3.10-slim`
*   **일반 이미지 vs slim 이미지**: 일반 `python:3.10` 이미지에는 우분투 환경의 온갖 개발 라이브러리와 유틸리티가 들어 있어 크기가 **약 1GB**에 달합니다. 반면 `slim` 버전은 리눅스 핵심 도구와 파이썬만 남기고 모두 걷어내어 용량을 **120MB 안팎**으로 다이어트한 것입니다.
*   **선택 이유**: AWS Free Tier가 제공하는 서버 디스크 크기(30GB)와 램 용량(1GB)은 매우 제한적입니다. 빌드 시간 단축과 자원 절약을 위해 반드시 경량 이미지를 사용해야 합니다.

### ② 레이어 캐싱(Layer Caching)과 COPY 순서의 비밀
*   도커는 빌드할 때 파일의 변화가 없는 라인은 이전의 빌드 결과물(캐시)을 재사용합니다.
*   만약 `COPY . .`를 먼저 실행하고 `pip install`을 실행하면, 파이썬 코드(`main.py` 등)의 글자 하나만 수정해도 도커는 파일이 변했다고 간주하여 **매번 용량이 크고 무거운 `pip install` 명령을 처음부터 다시 실행**합니다.
*   **해결책**: 변하지 않는 라이브러리 명세 파일인 `requirements.txt`만 먼저 복사하여 패키지를 설치한 후 코드 전체(`COPY . .`)를 복사하는 구조로 작성합니다. 이렇게 하면 코드가 수정되더라도 패키지 설치 단계는 **0.1초 만에 캐시로 복사**되어 빌드 속도가 극대화됩니다.

### ③ `--no-cache-dir` 옵션의 역할
*   `pip`는 기본적으로 패키지 재설치를 위해 로컬에 `.whl` 설치 파일을 캐싱해 둡니다. 하지만 일회성으로 완성되는 도커 이미지 안에는 이 캐시가 전혀 필요 없습니다.
*   `--no-cache-dir` 옵션을 붙여 캐시 파일 저장을 생략함으로써 이미지의 최종 용량을 추가로 수십~수백 MB 줄일 수 있습니다.

### ④ `--host 0.0.0.0` 바인딩의 이유
*   로컬 개발 환경에서는 웹서버를 보통 `localhost(127.0.0.1)`로 띄웁니다. 하지만 도커 컨테이너 내부에서 `127.0.0.1`로 띄우면, 오직 **컨테이너 내부의 자신**만 이 서버에 접속할 수 있고 컨테이너 밖(도커 네트워크 내의 다른 백엔드 컨테이너 등)에서는 접속이 완벽히 차단됩니다.
*   `0.0.0.0`은 "모든 네트워크 인터페이스(외부 가상 IP 포함)로부터 들어오는 무전 통신을 다 수락하겠다"는 선언입니다. 도커 배포 환경에서는 반드시 이를 지정해야 컨테이너 간 통신이 가능합니다.
